    Lesson 3 — The dashboard controller and the manager’s overview

    The sidebar’s Dashboard link points at /dashboard. Right now that route shows a placeholder page that says “Coming in Module 5.” This lesson lands the real thing — at least for managers.

    The lesson does two things:

    1. Build the dashboard controller that picks the right page component based on the current user’s role. One URL (/dashboard), three different pages depending on who you’re signed in as.
    2. Build the manager’s dashboard fully. Stat strip showing the operation’s scale, national map covering all 9 depots and 340 service areas, side panel with recent activity and busiest regions.

    Lesson 4 builds the dispatcher’s and field officer’s dashboards using the same controller. Lesson 5 wraps the module with the role switcher.

    The controller

    The dashboard controller’s job is small: identify the current user’s role and render the appropriate page component.

    Open app/controllers/dashboard_controller.rb. The chassis shipped a placeholder; we replace it with role dispatching:

    class DashboardController < ApplicationController
  def index
    @page_component = case Current.user.role
                      when "manager"       then Dashboards::ManagerDashboard
                      when "dispatcher"    then Dashboards::DispatcherDashboard
                      when "field_officer" then Dashboards::FieldOfficerDashboard
                      end
  end
end

    The action sets @page_component to a class — the actual page component that should render for this user. The view template instantiates and renders it:

    app/views/dashboard/index.html.erb
    1

    <%= render @page_component.new %>

    That’s the whole controller. One line per role mapping it to a page component class. The view template is two words.

    The pattern is honest: the controller decides what gets rendered; each page component decides how it renders. Splitting the responsibility cleanly means each role’s dashboard can grow as complex as it needs without the controller knowing or caring.

    The Dashboards namespace

    Page components for each role live in app/components/dashboards/. Create the directory:

    1

    mkdir app/components/dashboards

    Three files will go here over the next two lessons:

    • app/components/dashboards/manager_dashboard.rb (this lesson)
    • app/components/dashboards/dispatcher_dashboard.rb (Lesson 4)
    • app/components/dashboards/field_officer_dashboard.rb (Lesson 4)

    The namespace prefix isn’t strictly necessary — they could all live flat in app/components/. The grouping helps with navigation as the dispatch deck grows: by the end of Module 9 you’ll have a dozen or more page components, and finding “all the dashboards” is easier when they’re folded together.

    class Dashboards::ManagerDashboard < Components::Base
  def view_template
    div(class: "h-full flex flex-col bg-bg") do
      render Components::PageHeader.new(
        title:    "Operations dashboard",
        subtitle: "National overview of all dispatch activity",
        breadcrumb: ["Dashboard"]
      )

      render_stat_strip

      div(class: "flex-1 min-h-0") do
        render Components::MapPanelLayout.new do |l|
          l.map   { render Components::ServiceAreaMap.new }
          l.panel { render_side_panel }
        end
      end
    end
  end

  private

  def render_stat_strip
    div(class: "grid grid-cols-2 md:grid-cols-4 gap-4 px-8 py-5 bg-surface border-b border-border") do
      stat_tile("Depots",         Depot.count,                                     "across the country")
      stat_tile("Service areas",  ServiceArea.count,                               "ABS SA3 boundaries")
      stat_tile("Field officers", User.where(role: "field_officer").count,         "deployed nationally")
      stat_tile("Active jobs",    Job.where.not(status: %w[complete cancelled]).count, "currently in flight")
    end
  end

  def stat_tile(label, value, sublabel)
    div(class: "flex flex-col") do
      div(class: "text-3xl font-semibold text-text tabular-nums") { value.to_s }
      div(class: "text-sm text-text mt-0.5") { label }
      div(class: "text-xs text-text-muted") { sublabel }
    end
  end

  def render_side_panel
    div(class: "p-6 space-y-8") do
      render_recent_activity
      render_busiest_regions
    end
  end

  def render_recent_activity
    section do
      h2(class: "text-sm font-semibold text-text uppercase tracking-wider mb-3") { "Recent activity" }
      ul(class: "space-y-3") do
        recent_jobs.each do |job|
          li(class: "flex items-start justify-between gap-3 pb-3 border-b border-border last:border-0 last:pb-0") do
            div(class: "min-w-0") do
              div(class: "text-sm text-text font-medium truncate") { job.customer.name }
              div(class: "text-xs text-text-muted truncate") { "#{job.service_area.name}" }
            end
            div(class: "text-xs text-text-muted shrink-0 tabular-nums") do
              relative_time(job.completed_at)
            end
          end
        end
      end
    end
  end

  def render_busiest_regions
    section do
      h2(class: "text-sm font-semibold text-text uppercase tracking-wider mb-3") { "Busiest regions" }
      ol(class: "space-y-2") do
        busiest_regions.each_with_index do |region, idx|
          li(class: "flex items-center gap-3") do
            div(class: "text-sm text-text-muted tabular-nums w-5") { (idx + 1).to_s }
            div(class: "flex-1 min-w-0") do
              div(class: "text-sm text-text truncate") { region[:name] }
            end
            div(class: "text-sm text-text-muted tabular-nums") { region[:count].to_s }
          end
        end
      end
    end
  end

  def recent_jobs
    Job.where(status: "complete")
       .where.not(completed_at: nil)
       .order(completed_at: :desc)
       .limit(5)
       .includes(:customer, :service_area)
  end

  def busiest_regions
    ServiceArea
      .joins(:jobs)
      .group("service_areas.id", "service_areas.name")
      .select("service_areas.name AS name, COUNT(jobs.id) AS job_count")
      .order("job_count DESC")
      .limit(5)
      .map { |sa| { name: sa.name, count: sa.job_count } }
  end

  def relative_time(time)
    return "" unless time
    seconds = Time.current - time
    case seconds
    when 0..3600           then "#{(seconds / 60).to_i}m ago"
    when 3601..86_400      then "#{(seconds / 3600).to_i}h ago"
    when 86_401..604_800   then "#{(seconds / 86_400).to_i}d ago"
    else                        time.strftime("%-d %b")
    end
  end
end

    A few details worth pausing on.

    The page header. Reuses Components::PageHeader — same component used on the Lab page back in Module 4. Consistency matters here; every page in the dispatch deck has the same header treatment.

    The stat strip. Four tiles in a responsive grid (two columns on mobile, four on desktop). Each tile shows a label, a number, and a subdued sublabel. The actual numbers come from straightforward ActiveRecord queries — Depot.count, ServiceArea.count, etc. With our seed, these resolve to 9, 340, 30, and ~300 respectively.

    Components::ServiceAreaMap is the component we built in Module 4. It already shows all 340 service areas plus all 9 depots — exactly what the manager wants to see. No changes needed to the map component itself.

    The recent activity list. Five most recently completed jobs, with the customer name, service area, and a relative timestamp. The query uses includes(:customer, :service_area) to avoid N+1 — five jobs would otherwise produce ten extra queries (one for each customer and service area).

    The busiest regions list. Top five service areas by job count. The ST_* queries we’ll write in Module 6 will be more interesting; this one is plain SQL with a GROUP BY and ORDER BY. The map could later highlight these specifically; for Module 5 they’re just a list.

    relative_time. A small helper that produces “5m ago”, “3h ago”, “2d ago”, or a fixed date for older items. Not production-grade (no localisation, no future dates) but adequate for a dashboard’s needs.

    Activity 1 — Visit the manager’s dashboard

    Sign in as the manager (manager@vera.test / password). The Dashboard link in the sidebar is now active.

    You should see:

    • The page header with “Operations dashboard” and breadcrumb
    • The stat strip showing 9 depots, 340 service areas, 30 field officers, and your current active-jobs count
    • The map filling the left side, showing all of Australia with 9 red depot pins and 340 SA3 boundary outlines
    • The side panel on the right with two sections — Recent activity (5 completed jobs with timestamps) and Busiest regions (5 SA3s by job volume)

    Hover over an SA3 polygon — it highlights, just like in the Lab page. The same Components::ServiceAreaMap is doing the work; the dashboard is just composing it into the larger layout.

    Click a depot pin — the popup we built in Module 4 Lesson 5 appears with the depot’s name and code. All the Module 4 work is reusable now, in production-shaped contexts.

    Activity 2 — Sign in as a different role

    Sign out and back in as dispatcher@vera.test. Visit /dashboard. You should see… an error.

    The DashboardController tries to render Dashboards::DispatcherDashboard but that component doesn’t exist yet. Lesson 4 fills it in.

    For now, you can verify the role dispatching works by signing in as the manager again — the manager’s dashboard renders correctly. The pattern is in place; the other two pages arrive next lesson.

    Where this leaves us

    The manager has a working dashboard. They sign in, land on a page that shows the operation at a glance, and can begin to explore through the map and side panel.

    The infrastructure for the other two roles is there too — the controller dispatches correctly, the namespace is set up, the layout pattern is established. Lesson 4 builds out the dispatcher’s regional view (their state, their job list) and the field officer’s personal view (their patch, their jobs).
