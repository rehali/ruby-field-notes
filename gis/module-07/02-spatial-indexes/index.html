    How Spatial Indexes Actually Work

    By this point in the tutorial, every query you’ve written has been fast. That’s about to change.

    Up to now we’ve worked with the medium seed — about 50,000 jobs across Australia, weighted toward the cities. PostgreSQL handles that in milliseconds without breaking a sweat. The reason isn’t that PostgreSQL is magic. It’s that 50,000 rows is small. A sequential scan of 50,000 rows on modern hardware completes before you finish reading the EXPLAIN output.

    This chapter is where we step up to half a million jobs and run into the wall. Then we’ll punch through the wall by adding a spatial index — and along the way you’ll learn what’s actually inside one of those indexes, why they work the way they do, and where they don’t help.

    This isn’t a “indexes make queries faster” chapter. You already know that. This is a chapter about how PostGIS finds things in space, and what that means for the queries you write.

    What you’ll do

    You’ll generate the large seed dataset, watch a query crawl, and then make it fly with one CREATE INDEX statement. Along the way we’ll dissect the index — what it stores, how it’s organised, why it works. We’ll find a query that should benefit from the index but doesn’t, and understand why. And we’ll briefly compare GiST against SP-GiST so you know there’s more than one tool.

    By the end of this chapter you’ll have a working mental model of spatial indexing, not just a recipe.

    Step up to the large seed

    The medium seed has 50,000 jobs. The large seed has 500,000 — a year of dispatch operations across Australia. We need that volume because spatial indexing only matters at scale. Below maybe 100,000 rows, sequential scan is competitive. Above that, the difference becomes brutal.

    Activity 1 — Run the large seed

    In your project root:

    1

    bin/rails fieldflow:seed[large]

    The seed drops existing jobs and regenerates 500,000 against the existing service areas, technicians, and customer sites. It uses bulk inserts and should complete in 60–90 seconds depending on your machine. While it runs, read on.

    The seed weights job density by SA3 population, so most jobs land in the eastern capitals — Sydney, Melbourne, Brisbane, Perth, Adelaide — with thinner distributions through regional areas. A few jobs land in genuinely remote SA3s, just enough that the data looks plausible at country zoom. The script also varies job types, durations, and timestamps to look realistic.

    The query that motivates everything

    Let’s ask a simple operational question: what jobs happened within 5 km of the Brisbane CBD depot in the last 30 days?

    Open bin/rails dbconsole and try this:

    SELECT j.id, j.job_type, j.completed_at, jc.location
FROM jobs j
JOIN customer_sites jc ON jc.id = j.customer_site_id
WHERE j.completed_at >= NOW() - INTERVAL '30 days'
  AND ST_DWithin(
    jc.location::geography,
    (SELECT location::geography FROM depots WHERE name = 'Brisbane CBD'),
    5000
  )
ORDER BY j.completed_at DESC
LIMIT 100;

    This is a real business query — “what did we do near this depot recently?” The dispatcher would run it dozens of times a day.

    It returns. Not instantly. Maybe two or three seconds, maybe more. Long enough that you notice. In a web app under load, that’s a problem.

    That Seq Scan is what’s killing us. Two of them, in fact:

    • A sequential scan on jobs filtering by completed_at. Examines all 500,000 rows, keeps about 50,000 (one month of data). Slow but tolerable.
    • A sequential scan on customer_sites filtering by ST_DWithin(...). Examines all 100,000 customer sites, keeps about 8,000 within 5 km of the depot. This is the spatial scan, and it’s the bottleneck.

    For each of those 100,000 customer sites, PostgreSQL is computing a precise geographic distance from the depot. That’s a lot of geometric arithmetic to throw away most of the results.

    We can fix the second one with a spatial index. The first one we’ll come back to at the end of the chapter.

    The fix 

    1
2

    CREATE INDEX customer_sites_location_idx
  ON customer_sites USING GIST (location);

    That’s it. One statement. Try it now.

    But first: what just happened? What’s actually inside that index?

    What’s in a spatial index

    If you’ve worked with PostgreSQL before, you’ve probably created B-tree indexes — the default. A B-tree index for an integer column is a sorted tree of values: looking something like 1, 2, 3, 4 … organised so that “find all rows where id = 5043” or “find all rows where id BETWEEN 1000 AND 2000” can be answered without scanning the whole table.

    A spatial index can’t be a B-tree, because points in space don’t have a natural sort order. You can sort numbers — 5 comes before 7. You can’t sort coordinates — does (5, 7) come before or after (7, 5)? There’s no answer that makes sense for spatial queries.

    What PostGIS uses instead is a GiST index: Generalized Search Tree. GiST is a framework for tree-based indexes over types that don’t sort linearly. PostGIS plugs into it to provide a spatial index. The structure is roughly an R-tree — the standard spatial data structure since 1984, when Antonin Guttman introduced it.

    Here’s what an R-tree actually stores:

    Bounding boxes. Not the geometry itself. Not the polygon’s exact shape. Just the smallest rectangle (axis-aligned) that contains the geometry. For a point, that’s a degenerate rectangle. For a polygon, it’s the rectangle from the polygon’s minimum X/Y to its maximum X/Y. For a line, the rectangle that contains all the line’s vertices.

    Hierarchically grouped. The index isn’t a flat list of bounding boxes. It’s a tree where leaves contain individual geometries’ bounding boxes, and internal nodes contain bounding boxes that enclose all their children’s bounding boxes.

    You can see this for yourself. PostgreSQL exposes the index’s storage size:

    1

    SELECT pg_size_pretty(pg_relation_size('customer_sites_location_idx'));

    Probably something like 5760 kB. The customer_sites table has 100,000 rows; the index that covers them is 5–6 MB. Not nothing, but small compared to the 50+ MB the table itself uses. The index doesn’t store geometries — only their bounding boxes, organised hierarchically.

    Why this works (and where it fails)

    When you write ST_DWithin(geom, target, 5000), the query planner does something clever:

    1. It calls ST_Expand(target, 5000) — building a bounding box around the target that’s 5km bigger in each direction.
    2. It uses the index to find every row whose bounding box overlaps that expanded box. This is fast — it’s a tree traversal, eliminating most of the table’s rows in the process.
    3. For the rows the index returns, it computes the exact ST_DWithin distance. This is the accurate test, but now we’re only running it on (say) a few hundred candidates, not 100,000.

    That two-phase pattern — index narrows down the candidates, then exact computation on the survivors — is the core of how spatial indexing works. The index’s job is to be conservative: never miss a row that might match, but happily include some that won’t. The exact computation filters out the false positives.

    This is also where things can go wrong. The index works for any query the planner can rewrite as “show me rows whose bounding box overlaps something.” But not every spatial query can be rewritten that way.

    Let’s look at one that can’t.

    1
2
3

    EXPLAIN ANALYZE
SELECT id FROM customer_sites
WHERE ST_Distance(location::geography, ST_MakePoint(153.025, -27.467)::geography) < 5000;

    This looks equivalent to ST_DWithin. It asks for customer sites within 5 km of a point. But check the plan:

    1
2

    Seq Scan on customer_sites  (cost=0.00..3567.45 rows=33333 width=8)
  Filter: (ST_Distance(...) < 5000::double precision)

    Sequential scan. The index isn’t being used. Why?

    Because ST_Distance(...) < 5000 doesn’t tell the planner anything spatial. From the planner’s perspective, this is just a function returning a double, compared to 5000. The function happens to compute a geographic distance, but the planner can’t see inside it. There’s nothing to rewrite as a bounding box query.

    ST_DWithin, by contrast, is explicitly indexable. PostGIS has registered it with the planner so that when the planner sees ST_DWithin(col, target, distance) and col has a spatial index, it knows to use the index.

    This is one of the most important practical things to know about PostGIS:

    Use ST_DWithin, not ST_Distance, for proximity queries.

    ST_DWithin(geom, target, distance) uses the spatial index. ST_Distance(geom, target) < distance does not, and produces a sequential scan no matter how many indexes you have.

    Other indexable functions: ST_Intersects, ST_Contains, ST_Within, ST_Crosses, ST_Overlaps, ST_Touches, ST_Equals. Anything that can be expressed as “do these bounding boxes overlap” gets the index.

    This is the kind of mistake that’s easy to make and very hard to debug after the fact. A query is slow; you check the index exists; the index does exist; you don’t realise that the function you’re using doesn’t trigger it. Knowing the indexable functions saves hours.

    A second index type — SP-GiST

    GiST is the default and works for almost everything. But there’s a second option worth knowing about: SP-GiST (Space-Partitioning GiST).

    The difference:

    • GiST uses overlapping bounding boxes. Two adjacent points might appear in two different tree branches that overlap each other. This is fine in practice, but it means the tree is slightly less efficient for some queries.
    • SP-GiST uses non-overlapping space partitioning, like a quadtree or k-d tree. Each region of space appears in exactly one branch. Queries can be slightly faster on point-only data with no overlap.

    SP-GiST works best when:

    • All your geometries are points (no polygons, lines, or mixed types).
    • You don’t care about exact-equals queries (ST_Equals) — SP-GiST doesn’t support them.
    • You’re optimising heavily and the GiST index is already a hot spot.

    For our customer_sites table, which is points only, SP-GiST is a candidate. Let’s compare.

    Activity 4 — Compare GiST and SP-GiST

    Drop the GiST index and create an SP-GiST index instead:

    1

    DROP INDEX customer_sites_location_idx;
CREATE INDEX customer_sites_location_idx
  ON customer_sites USING GIST (location);

    In our testing on this dataset:

    • GiST index size: ~5.7 MB. Query time: ~310 ms.
    • SP-GiST index size: ~3.2 MB. Query time: ~290 ms.

    A small win — about 6% faster, half the size. For a hot table with billions of rows, that adds up. For a hundred thousand rows, GiST is fine.

    In practice, start with GiST. It works for everything. Reach for SP-GiST only when you’ve measured a need and your data is point-only.

    The other slow part — temporal indexing

    We focused on the spatial part, but our original query had two slow scans. The other was on jobs, filtering by completed_at >= NOW() - INTERVAL '30 days'. The query planner is doing a sequential scan over 500,000 rows to find the recent ones.

    That’s a regular B-tree problem, not a spatial one. The fix is a B-tree index on completed_at:

    1

    CREATE INDEX jobs_completed_at_idx ON jobs (completed_at);

    Run the EXPLAIN ANALYZE one more time. The Brisbane CBD query should now drop from 312 ms to under 50 ms — both halves of the query are now indexed.

    This is worth flagging because spatial Rails apps often have mixed-mode queries like this one, joining time and space. Both kinds of indexes typically need to coexist. Forgetting either one leaves the query slow even with the other in place.

    Re-run the original query. Note the new execution time.

    What to take away

    A few things from this chapter that will pay back many times over the rest of the tutorial:

    Spatial indexes store bounding boxes, not geometries. The index narrows down candidates by bounding-box overlap; the exact spatial test runs on the survivors. This is why indexes work — and also why their effectiveness depends on your data’s spatial distribution.

    Use indexable functions. ST_DWithin, ST_Intersects, ST_Contains, ST_Within, ST_Crosses, ST_Overlaps, ST_Touches. Avoid ST_Distance(x, y) < n patterns — they don’t use the index and the planner can’t rewrite them.

    Mixed-mode queries need mixed-mode indexes. A query that filters by both space and time needs a spatial index AND a B-tree index. Adding only one of them leaves the slowdown in place.

    GiST is the default; SP-GiST is for tuning. Don’t reach for SP-GiST until you’ve measured a need and your data is point-only.

    Most importantly: when you write a spatial query, think about what bounding boxes the index can use to narrow it down. If you can’t see how the index would help, the index won’t help. Rewriting the query to be index-friendly is usually possible — and the right move.
