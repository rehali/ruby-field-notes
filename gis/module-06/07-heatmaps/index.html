    Lesson 7 — Heatmaps

    The dispatch deck has rendered points three ways now: individual circles (Module 5 jobs), clustered bubbles (Module 6 Lesson 2), and aggregated polygons (Lessons 4 and 5’s choropleths). Each fits a different question. Individual circles when each point is the unit of interest. Clusters when the interest is “how many roughly here?” Choropleths when the unit of interest is the region, not the point.

    This lesson introduces a fourth: heatmaps. MapLibre’s type: :heatmap layer renders point data as a continuous gradient cloud — denser regions glow brighter, sparser regions fade out. There are no individual features, no discrete buckets, no polygon boundaries. Just density, spatially smooth.

    Heatmaps fit a different question from the others:

    • Where is concentration? — the gradient is the answer
    • Individual data points are interesting only as contributors, not as features in themselves
    • The map’s purpose is to reveal pattern, not to enable interaction with specific records

    For our data, the natural question is where work happens. Where do jobs cluster geographically? It’s a strategic question — a manager wondering “where are we busiest?” or “which regions deserve more capacity?” The answer is a continuous geographic truth, not a discrete count per administrative region.

    This lesson builds a job-density heatmap — a fourth manager view that visualises the dispatch deck’s complete operational footprint at the density-pattern level.

    But there’s a tension to navigate before any code. Heatmaps need density to read clearly. A few thousand points scattered across a continent produce a barely-visible signal. Twenty thousand points start to register. A hundred thousand points produce a strong, gradient-rich heatmap.

    Our customers are too sparse — 3,000 across Australia, with maybe 300-400 in Sydney metro. A customer-density heatmap would barely glow. Jobs are denser — 500K total over two years. That’s heatmap-friendly density.

    But shipping 500K full GeoJSON features to the browser is its own problem — each feature is over 100 bytes of JSON, so the payload becomes 60-70MB. Even gzipped that’s 15-20MB. The browser would parse for seconds before MapLibre even started rendering.

    This lesson’s substance is solving both problems together:

    • A compact wire format (MultiPoint geometry) that ships many points with minimal overhead
    • A two-layer caching strategy (server cache + browser cache) so the response builds rarely and serves quickly

    Together these turn a 500K-point heatmap into a fast, production-grade view.

    How heatmaps work

    MapLibre’s heatmap layer is conceptually simple but uses several paint properties working together to produce its characteristic look. The core idea: each point contributes a small radial “bump” of intensity to the rendered image; the bumps accumulate; a colour ramp maps accumulated intensity to RGB values.

    The four paint properties that matter:

    • heatmap_weight — how much each point contributes. Usually 1 (every point counts equally) but can vary by feature property.

    • heatmap_intensity — a global multiplier that scales the entire heatmap’s brightness. Often varies with zoom.

    • heatmap_radius — the pixel radius of each point’s bump. Smaller radius → tighter, more granular hotspots; larger radius → smoother, more abstract gradients. Often varies with zoom.

    • heatmap_color — a colour ramp applied to the accumulated intensity. Standard pattern: transparent at low intensity, ramping through cool colours, then through warm, peaking at white-hot.

    Three of these (intensity, radius, color) are typically expressed as zoom-dependent expressions. That’s what gives heatmaps their characteristic ability to read at any zoom: zoomed out, broad patterns; zoomed in, finer detail.

    MultiPoint — the compact wire format

    GeoJSON supports several geometry types. The most familiar is Point, which describes a single location:

    1

    { "type": "Point", "coordinates": [151.123, -33.123] }

    Wrapped as a Feature in a FeatureCollection — the standard pattern we’ve used everywhere — that point becomes:

  "type": "Feature",
  "id": 12345,
  "geometry": { "type": "Point", "coordinates": [151.123, -33.123] },
  "properties": {}
}

    About 110 bytes. The geometry itself is 50 bytes; the rest is Feature wrapping overhead.

    MultiPoint is a different geometry type that represents a collection of points as a single geometry:

  "coordinates": [
    [151.123, -33.123],
    [151.456, -33.456],
    [151.789, -33.789]
  ]
}

    Wrapped as a Feature, the entire 500K-point dataset becomes one Feature whose geometry is a MultiPoint:

    {
  "type": "FeatureCollection",
  "features": [{
    "type": "Feature",
    "geometry": {
      "type": "MultiPoint",
      "coordinates": [[lng, lat], [lng, lat], ...]
    },
    "properties": {}
  }]
}

    Each point is now around 20 bytes — just the comma-separated coordinate pair. 5-6× smaller payload than separate Features. At 500K points, that’s roughly 10MB raw / 3MB gzipped, instead of 60MB raw / 15MB gzipped.

    And MapLibre’s heatmap layer handles MultiPoint geometries natively. Each constituent point contributes to the density exactly as it would if it were a separate Point feature. The visualisation is identical; only the wire format is denser.

    The trade-off: with MultiPoint, the points lose individual identity. There’s no feature.id per point, no per-point properties, no clicking a specific point. That’s fine for a heatmap — heatmaps don’t expose individual features anyway. For visualisations where each point needs identity (jobs you can click to drill into), separate Point features remain the right choice.

    Caching — server side and browser side

    The query that builds the dataset reads 500K rows from the database and converts them into a coordinate array. That’s genuinely a second or two of work even with the GIST index helping. We don’t want to do it on every request.

    For data that changes slowly — and our heatmap data really does (jobs accumulate at <1% per week) — caching is the answer. There are several layers where caching can live; the strongest production pattern combines two of them.

    Server cache — the response data is cached in Rails' shared cache (memory, Redis, or Memcached). Subsequent requests skip the database query and the JSON-building work entirely; the controller just serves the cached structure.

    1
2
3

    Rails.cache.fetch("job_locations:multipoint", expires_in: 1.day) do
  build_locations
end

    fetch checks the cache first; if it’s a miss (or stale), it runs the block to compute the value, stores the result, and returns it. The next call within the cache window returns the stored value directly. The cache key needs to be unique to this dataset; expires_in sets the upper bound on staleness.

    Browser cache — HTTP cache headers tell the browser to remember the response and serve it locally on subsequent requests within the cache window.

    1

    expires_in 1.day, public: true

    This adds a Cache-Control: max-age=86400, public header to the response. The browser stores the response. The next time the user navigates to the heatmap page, the browser serves the data from local storage without contacting the server at all. Instant load, zero network round-trip.

    Together these cover all cases:

    • Browser has it cached → no network request. Instant.
    • Browser cache stale, server cache warm → server returns cached structure; ~10ms response.
    • Both caches cold (first request after expiry) → server rebuilds (~1-2s), caches it, returns. Subsequent requests hit the warm caches.

    The server cache also protects against the thundering herd — if many users arrive simultaneously after the cache expires, only the first one rebuilds; everyone else waits briefly and then hits the warm cache.

    On invalidation: time-based expiry (expires_in: 1.day) is the simplest pattern, and right for our case. The heatmap drifts so slowly that a day-stale view is operationally indistinguishable from a fresh one. More rigorous patterns exist — versioned cache keys that include something like Job.maximum(:updated_at).to_i so the key changes whenever data changes — but they add complexity. For low-drift data like ours, simple time expiry is the right call.

    The 1-day expiry is a comfortable choice; the data changes <1% per week so even a one-week cache would be operationally acceptable.

    module JobLocations
  extend self

  CACHE_KEY = "job_locations:multipoint".freeze
  CACHE_TTL = 1.day

  # Returns a single-feature FeatureCollection whose geometry is
  # a MultiPoint of all job locations. Cached for 1 day in the
  # Rails cache; rebuild cost is ~1-2 seconds at 500K points.
  def call
    Rails.cache.fetch(CACHE_KEY, expires_in: CACHE_TTL) do
      build_locations
    end
  end

  # Force-rebuild the cache. Useful for tests or after bulk data
  # imports that should be reflected immediately.
  def refresh
    Rails.cache.delete(CACHE_KEY)
    call
  end

  private

  def build_locations
    rows = ActiveRecord::Base.connection.select_all(<<~SQL).rows
      SELECT ST_X(location) AS lng, ST_Y(location) AS lat
      FROM jobs
    SQL

    {
      type: "FeatureCollection",
      features: [{
        type: "Feature",
        geometry: {
          type:        "MultiPoint",
          coordinates: rows.map { |lng, lat| [lng, lat] }
        },
        properties: {}
      }]
    }
  end
end

    A few things worth understanding.

    ST_X(location) and ST_Y(location) extract the longitude and latitude as raw floats. The query returns ~500K rows of two floats each — the smallest possible representation Postgres can produce.

    The rows.map { |lng, lat| [lng, lat] } is identity-ish — it just turns the row tuples into 2-element arrays. The select_all(...).rows gives us arrays already; the map just makes the structure explicit and converts to the form the JSON serialiser will produce as [lng, lat] pairs.

    Rails.cache.fetch(KEY, expires_in: TTL) do ... end is the canonical caching pattern. The block runs only when the cache is cold; otherwise the cached value is returned directly.

    refresh is a small convenience for invalidation. The seed task could call JobLocations.refresh after rebuilding job data, ensuring the cache reflects the new state without waiting for the daily expiry.

  def locations

    render json: JobLocations.call
  end
end

    The expires_in here is a Rails controller method that sets HTTP cache headers, not the same as the cache method’s expires_in keyword. (Confusing naming.) expires_in 1.day, public: true produces:

    Cache-Control: max-age=86400, public

    public: true is important — it permits shared caches (like CDNs) to cache the response too. Without it, only individual browsers would cache. For data that’s identical across all manager users, public caching is correct.

    class Components::Reports::JobDensityMap < Components::Base
  prop :id,     String, default: -> { "job-density-map" }
  prop :height, String, default: -> { "100%" }

  def view_template
    render Vera::Map.new(id: @id, height: @height, style: :voyager,
                         zoom_indicator: :bottom_left) do |m|
      m.control :navigation
      m.control :fullscreen
      m.control :scale, unit: :metric

      add_heatmap(m)
    end
  end

  private

  def add_heatmap(m)
    m.source :jobs,
             url:        "/api/jobs/locations.json",
             fit_bounds: { padding: 20 }

    m.layer :job_heatmap, source: :jobs, type: :heatmap,
            paint: heatmap_paint
  end

  # The four heatmap paint properties working together. Each is
  # a zoom-dependent expression so the visual scale stays sensible
  # from national zoom (4) through metro zoom (12).
  def heatmap_paint
    {
      # Each point counts equally. A weighted heatmap would vary
      # this by feature property — but our MultiPoint geometry
      # doesn't carry per-point properties, so weights are the
      # same for every point.
      heatmap_weight: 1,

      # Global brightness multiplier. Higher zoom levels need
      # more amplification because individual points are spread
      # across more pixels and would otherwise fade to invisible.
      heatmap_intensity: [
        "interpolate", ["linear"], ["zoom"],
        4, 1,        # national zoom: baseline
        12, 3        # metro zoom: 3x amplification
      ],

      # Pixel radius of each point's contribution. At national
      # zoom we want broader bumps so density patterns smooth
      # together; at metro zoom we want tighter bumps so individual
      # neighbourhoods become visible.
      heatmap_radius: [
        "interpolate", ["linear"], ["zoom"],
        4, 6,         # national zoom: small contribution per point
        12, 30        # metro zoom: larger contribution per point
      ],

      # The colour ramp. Density values normalised to 0..1 by
      # MapLibre; we map that range through cool-to-warm-to-white.
      # Transparent at the low end so empty regions show the
      # basemap clearly through the heatmap layer.
      heatmap_color: [
        "interpolate", ["linear"], ["heatmap-density"],
        0,    "rgba(0, 0, 255, 0)",
        0.1,  "rgba(0, 96, 255, 0.4)",
        0.3,  "rgba(0, 192, 255, 0.6)",
        0.5,  "rgba(127, 255, 127, 0.7)",
        0.7,  "rgba(255, 255, 0, 0.8)",
        0.9,  "rgba(255, 96, 0, 0.9)",
        1.0,  "rgba(255, 255, 255, 0.95)"
      ],

      # Fade the heatmap into the basemap at very high zoom levels.
      # When the user zooms in close enough to see individual
      # streets, the heatmap becomes less informative than seeing
      # the basemap clearly.
      heatmap_opacity: [
        "interpolate", ["linear"], ["zoom"],
        4,  0.85,    # mostly opaque at national zoom
        14, 0.4      # more transparent at neighbourhood zoom
      ]
    }
  end
end

    A walkthrough of the substantive bits.

    heatmap_weight: 1 — every point contributes the same. Because we used MultiPoint geometry to ship the points, there are no per-point properties to vary weights by anyway. If we had wanted weighted heatmaps (urgent jobs counting more, say), we’d have shipped Point features with priority properties and used ["match", ["get", "priority"], "urgent", 5, ...] here.

    heatmap_intensity varying with zoom — at zoom 4 (national view), the heatmap renders at baseline brightness. At zoom 12 (city view), brightness multiplies by 3 because individual points are now spread across many more pixels and would fade to invisibility without amplification. The interpolate ["linear"] ["zoom"] expression interpolates linearly between the named stops, smoothly transitioning as the user zooms.

    heatmap_radius varying with zoom — at zoom 4, each point contributes a 6-pixel bump (small, so density patterns smooth together visually). At zoom 12, each contributes a 30-pixel bump (larger, so individual neighbourhoods read as warm spots). This is the property most responsible for “heatmap behaving sensibly across zoom levels.”

    heatmap_color with seven stops — the classic cool-to-warm ramp. The first stop at 0 is transparent (so empty regions show the basemap unobstructed); subsequent stops ramp through blues, greens, yellows, oranges, peaking at white at maximum density.

    heatmap_opacity varying with zoom — at high zoom levels, the heatmap fades. Otherwise it would obscure the basemap’s street-level detail just when the user is most likely to want to see it. By zoom 14 the heatmap is at 40% opacity, allowing streets and labels to read through.

    Look what just happened

    Sign in as the manager. The Reports section now has four items. Click Job Density.

    The first load takes a moment — the cache is cold, the server is querying 500K rows and assembling the MultiPoint payload. Around 1-2 seconds. The browser receives the response and MapLibre starts rendering.

    The map fills with a national heatmap. Sydney metro glows bright white at its centre — hundreds of thousands of jobs over two years stack into intense density. Melbourne and Brisbane similar, glowing strongly. Adelaide, Perth, Newcastle, Canberra each appear as warm distinct spots. The corridor between Sydney and Newcastle shows a long warmer band. Tasmania has a small Hobart hotspot. The outback reads as transparent — few jobs means little heat.

    Refresh the page. Instant load — the browser has the response cached locally; no network request happens at all. The user sees the heatmap immediately.

    Click around to other reports and back to Job Density. Still instant — browser cache is doing its work.

    Zoom in toward Sydney. The single white-hot blob resolves into multiple distinct hotspots — the CBD, eastern suburbs, inner west, north shore each become individually visible.

    Zoom in further toward, say, Bondi. The hotspot resolves into even finer texture. By zoom 14 the heatmap fades and the basemap’s street labels become readable; you can see the actual streets the job density follows.

    This is the heatmap’s defining behaviour. The same dataset reads at multiple scales — broad geographic patterns, regional distinctions, local clusters, neighbourhood detail — without any code change between zoom levels.

    How heatmaps differ from clustering

    Both visualise dense point sets. The differences matter for when each fits.

    Clustering produces discrete features at any zoom. Cluster bubbles have counts; clusters split into smaller clusters as the user zooms; eventually individual points emerge. The interaction model is “click a cluster to drill in.” The data is exposed as data.

    Heatmaps produce continuous gradients. No discrete features, no counts, no click target. The interaction model is “look and read the pattern.” The data is exposed as visual texture.

    Use clustering when:

    • The user might want to see, click, or count individual points
    • Each point matters as a unit (a job, a customer interaction)
    • The map is part of an operational workflow

    Use heatmaps when:

    • Density is the question; individual points are noise
    • The user wants to perceive pattern, not act on records
    • The map is part of strategic analysis or exploration

    For the dispatch deck: dispatcher and FO maps use clustering because each job and each FO matters individually. The manager’s job-density map uses a heatmap because the question is geographic concentration, not which job is which.

    What this introduced

    Several patterns worth carrying forward:

    The heatmap layer type and its paint expressions. Four properties (weight, intensity, radius, color) plus opacity — each commonly zoom-dependent. The pattern of ["interpolate", ["linear"], ["zoom"], ...] is fundamental to making heatmaps behave sensibly at scale.

    Zoom-dependent paint expressions in general. Not just for heatmaps. Any visual property can vary with zoom — circle radii, line widths, label sizes, fill opacities. The same interpolate-by-zoom pattern applies. This is how MapLibre maps “do the right thing” across scale ranges.

    MultiPoint geometry for compact point payloads. When points need no per-point identity, a single MultiPoint Feature ships them at 5-6× less wire size than separate Point Features. Trade-off: you lose feature identity, so this pattern is only right for visualisations like heatmaps that don’t expose individual points.

    Two-layer caching for slowly-changing data. Server cache (Rails.cache) protects the build cost; browser cache (HTTP headers) eliminates the network round-trip on the user’s side. Together they make a 500K-point heatmap as fast to load as a static image, after the first visit.

    Heatmaps as a strategic tool. Operationally, heatmaps answer “where is concentration” questions — siting, penetration, gap analysis. They don’t answer “which records” questions. Knowing which paradigm fits a given operational question is the real skill.
