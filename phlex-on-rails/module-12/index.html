    Module 12 — Finishing Touches

    6 lessons · Finishing Touches · KanbanFlow complete
    We complete KanbanFlow with card details, search, error handling, and deployment — then step back and reflect on what building with Phlex actually felt like.

    Before we start

    KanbanFlow is functionally complete. Users can sign in, create boards, invite collaborators, move cards in real time, and switch themes. What remains is the work that separates a working app from a finished one — card details, search, polish, error handling, and deployment.

    This module also gives us space to reflect. By the end of Lesson 5 we will have an honest assessment of what Phlex made easy, what it made hard, and where the ecosystem is heading. That reflection is as important as the code.

    # app/views/cards/show.rb
class Views::Cards::Show < Views::Base
  def page_title = @card.title

  def initialize(card:, policy:)
    @card   = card
    @policy = policy
  end

  def view_template
    Breadcrumb() do |b|
      b.item "Boards",           url: boards_path
      b.item @card.column.board.name, url: board_path(@card.column.board)
      b.item @card.column.name
      b.item @card.title
    end

    div(class: "max-w-3xl mt-6 space-y-6") do
      render_header
      render_description
      render_metadata
      render_assignees
    end
  end

  private

  def render_header
    div(class: "flex items-start justify-between gap-4") do
      h1(class: "text-2xl font-bold text-text flex-1") { @card.title }
      if @policy.edit?
        a(href: edit_column_card_path(@card.column, @card),
          class: "text-sm text-text-muted hover:text-text") { "Edit" }
      end
    end
  end

  def render_description
    div(class: "prose prose-sm max-w-none") do
      if @card.description.present?
        raw safe(render_markdown(@card.description))
      else
        p(class: "text-text-subtle italic") { "No description." }
      end
    end
  end

  def render_metadata
    div(class: "flex gap-6 text-sm") do
      if @card.label
        div do
          p(class: "text-text-muted text-xs mb-1") { "Label" }
          Badge(label: @card.label.capitalize, variant: label_variant)
        end
      end

      if @card.due_date
        div do
          p(class: "text-text-muted text-xs mb-1") { "Due date" }
          p(class: due_date_classes) { @card.due_date.to_fs(:long) }
        end
      end
    end
  end

  def render_assignees
    return if @card.assignees.empty?

    div do
      p(class: "text-text-muted text-xs mb-2") { "Assignees" }
      div(class: "flex gap-2") do
        @card.assignees.each do |user|
          Avatar(name: user.name, size: :sm)
        end
      end
    end
  end

  def render_markdown(text)
    Redcarpet::Markdown.new(
      Redcarpet::Render::HTML.new(safe_links_only: true),
      autolink: true, tables: true
    ).render(text)
  end

  def label_variant
    { "bug" => :danger, "feature" => :primary,
      "improvement" => :success, "design" => :warning,
      "research" => :default }.fetch(@card.label, :default).to_sym
  end

  def due_date_classes
    base = "font-medium"
    if @card.due_date < Date.today
      "#{base} text-danger"
    elsif @card.due_date == Date.today
      "#{base} text-warning"
    else
      "#{base} text-text"
    end
  end
end

    // app/javascript/controllers/search_controller.js
import { Controller } from "@hotwired/stimulus"

export default class extends Controller {
  static targets = ["input", "results"]
  static values  = { url: String }

  query() {
    clearTimeout(this.debounceTimer)
    this.debounceTimer = setTimeout(() => {
      const query = this.inputTarget.value.trim()
      if (query.length < 2) {
        this.resultsTarget.classList.add("hidden")
        return
      }
      this.fetch(query)
    }, 300)
  }

  fetch(query) {
    fetch(`${this.urlValue}?q=${encodeURIComponent(query)}`, {
      headers: { Accept: "text/vnd.turbo-stream.html" }
    })
    .then(r => r.text())
    .then(html => {
      this.resultsTarget.innerHTML = html
      this.resultsTarget.classList.remove("hidden")
    })
  }
}

    # app/controllers/search_controller.rb
class SearchController < ApplicationController
  def index
    @query   = params[:q].to_s.strip
    @results = search_results

    respond_to do |format|
      format.turbo_stream do
        render turbo_stream: turbo_stream.replace(
          "search-results",
          Views::Search::Results.new(results: @results, query: @query).call
        )
      end
      format.html do
        render Views::Search::Index.new(results: @results, query: @query)
      end
    end
  end

  private

  def search_results
    return [] if @query.length < 2

    boards = current_user.boards
                         .where("name ILIKE ?", "%#{@query}%")
                         .limit(5)

    cards  = Card.joins(column: :board)
                 .merge(current_user.boards)
                 .where("cards.title ILIKE ? OR cards.description ILIKE ?",
                        "%#{@query}%", "%#{@query}%")
                 .limit(10)

    { boards: boards, cards: cards }
  end
end

class Components::FilterBar < Components::Base

        field:    :label,
        options:  label_options,

        field:    :assignee,
        options:  assignee_options,

          class: "text-xs text-text-muted hover:text-text") { "Clear filters" }

    end
  end

  private

  def label_options
    Card::LABELS.map { |l| [l.capitalize, l] }
  end

  def assignee_options
    @board.members.map { |u| [u.name, u.id.to_s] }
  end
end

    Lesson 5 — Retrospective

    What Phlex made easy

    Testing. Every component and view is a plain Ruby object. Instantiate it, call it, assert the output. The test suite for Phlex::UI runs in under a second. In a traditional Rails app, testing views means controller specs, request specs, or system tests — all slower and more fragile. With Phlex, a component test is as fast as a model test.

    Composition. KanbanColumn renders KanbanCard which renders Dropdown which renders Link. Each level knows only about its own props. Changing Dropdown’s implementation doesn’t require touching KanbanCard. This is component composition working as intended — and it’s harder to achieve with ERB partials.

    Tailwind synergy. Every Tailwind class lives in one Ruby file, alongside the markup it styles. The Module 7 token system — bg-primary replacing bg-blue-600 — was only possible because components own their styles. In an ERB app, Tailwind classes scatter across dozens of templates.

    The explicit interface. prop :board, Board documents what a component needs. You can’t accidentally pass the wrong type — Literal raises immediately. In ERB, a partial’s interface is implicit: read the template to discover what variables it needs.

    Refactoring confidence. Rename a prop, move a component, change a slot — the test suite tells you immediately what broke. The explicit interface makes the blast radius of any change predictable.

    What Phlex made harder

    Prose-heavy views. A page with significant static text is more verbose in Phlex than ERB. Every string needs plain "...". A marketing page or documentation view that’s 80% static HTML is genuinely better expressed in ERB. The hybrid approach (Module 4, Lesson 2) exists precisely for this case.

    The learning curve. vanish, capture, raw safe, slot naming conflicts with Phlex HTML methods — these are real gotchas that took real time to discover and document. Modules 3 and 4 of this series are a testament to how much there is to learn before Phlex feels natural. ERB has a much shallower initial curve.

    The ecosystem. Phlex is younger than ERB and ViewComponent. Some Rails helpers don’t have adapters yet. Gems that render HTML assume ERB. You will occasionally hit a wall where something that “just works” in ERB requires extra effort in Phlex. This is improving rapidly but it’s real.

    Form builders. form_with in Phlex is workable — Module 6 proved that — but the form builder’s output helpers push to the buffer in ways that require understanding the output/value helper distinction. It’s not hard, but it’s not obvious.

    The honest verdict

    Phlex is the right choice for:

    • Greenfield apps where you control the full stack
    • Component libraries that need to be portable and testable
    • Teams comfortable with Ruby who want to stay in Ruby for views
    • Apps that will grow — the composition and testing benefits compound

    ERB is still the right choice for:

    • Existing apps where migration cost outweighs benefit
    • Prose-heavy views with little dynamic content
    • Teams who need to onboard quickly and can’t invest in the learning curve
    • Rapid prototypes where structure matters less than speed

    The hybrid approach — ERB for pages, Phlex for components — is not a compromise. It’s a legitimate architecture that lets you capture the benefits of Phlex where they matter most while avoiding the costs where they don’t.

    Where Phlex is heading

    The Phlex ecosystem is growing quickly. phlex-rails is actively maintained. Lookbook has first-class Phlex support. The community is building UI libraries, form helpers, and testing utilities. Phlex 2.x is significantly faster than 1.x and the performance story continues to improve.

    The most interesting direction is the component-as-API pattern — using prop declarations and Literal types to make components self-documenting in a way that tools can analyse. This is where Phlex diverges most from ERB: components are objects, and objects can be reasoned about by tooling in ways that templates cannot.

    Lesson 6 — Congratulations

    If you got here, you built this:

    KanbanFlow — a production-ready, real-time, multi-user Kanban board built entirely with Phlex and Rails 8.

    Let’s appreciate what that means:

    • 12 modules, 72 lessons of progressively more complex Phlex
    • A complete component library — Phlex::UI with 20+ components, Lookbook previews, Tailwind token theming, and dark mode
    • A real application — authentication, role-based access control, real-time collaboration via ActionCable, drag-and-drop, inline editing, search, and deployment with Kamal
    • Hard-won knowledge — every gotcha in this series was discovered by building the thing for real. The slot naming conflicts, the _Boolean defaults, the raw safe pattern, the form_with output helper distinction — you now know all of these because you’ve been through the journey

    What you have

    Phlex::UI is a portable component library. It’s not tied to KanbanFlow — take it to your next Rails project. The Tailwind token system means it adapts to any brand with CSS changes only.

    The patterns — explicit interfaces, slot-based composition, vanish for named slots, StimulusComponent, self-permitting forms — are not KanbanFlow-specific. They are transferable to any Phlex project.

    The judgment — you now know when to reach for Phlex and when to reach for ERB. When to use Morphs and when to use Frames. When to extract a component and when to leave it as a private method. This judgment is the most valuable thing the series can give you.
